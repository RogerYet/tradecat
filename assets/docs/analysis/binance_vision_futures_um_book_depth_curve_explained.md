# Binance Futures UM `bookDepth` 曲线：白话版（是什么 / 为什么 / 怎么用）

> 一句话先抓住：`bookDepth` 就是在某个时刻，问一句 **“离当前价 ±x% 的范围里，买盘/卖盘一共挂了多少量（depth）和值多少钱（notional）？”**  
> 它不是“价格”，是“缓冲垫厚度”；用来判断 **滑点会不会爆、冲击成本有多大、哪边更抗打**。

---

## 0) 这玩意到底在测什么（最小真相）

市场价格从第一性原理上只由一件事决定：

- 你买 = 吃掉卖盘（ask）挂单
- 你卖 = 吃掉买盘（bid）挂单

如果离当前价很近的地方挂单很多（“厚”）：

- 你吃一口，不会把价格推太远（滑点小）

如果离当前价很近的地方挂单很少（“薄”）：

- 你吃一口，就会把价格推很远（滑点大 / 风险大）

`bookDepth` 就是把复杂的 L2 订单簿压缩成一句可计算的话：

- **在 ±x% 的范围内，有多少“可吃的对手盘”（数量/金额）？**

---

## 1) 是什么（What）——每一列说人话

官方 CSV（Vision）列：`timestamp,percentage,depth,notional`

- `timestamp`：这一刻的盘口曲线时间（约定按 UTC 解析）。
- `percentage`：离 **mid（买一卖一中间价）** 的距离，百分比表示：
  - `-0.20%/-1%/-5%`：看 **买盘（bid）** 的缓冲垫（向下这段范围里能接多少货）
  - `+0.20%/+1%/+5%`：看 **卖盘（ask）** 的缓冲垫（向上这段范围里能出多少货）
- `depth`：在这个范围内累计可成交的 **数量**（Σ qty）。
- `notional`：在这个范围内累计可成交的 **金额**（Σ price * qty）。

同一个 `timestamp` 会出现多行：因为它会给固定的一组 `percentage` 点（例如 ±0.2/1/2/3/4/5%）。

---

## 2) 为什么有用（Why）——价格为什么会动（第一性原理）

你想赚钱，本质上要控制一件事：**你的交易成本（滑点/冲击）别把你的 alpha 吃光**。

`bookDepth` 直接告诉你：

- 你如果“现在”下一个市价单/激进限价单，会不会把价格打穿（冲击）？
- 你如果挂单做市，离 mid 很近的地方有没有足够“保护层”（被扫单风险）？
- 买卖两边哪边更厚（短周期压力更偏哪边）？

---

## 3) 怎么读（How to read）——一个具体例子（不用术语）

假设这一刻 mid≈100：

- `percentage=+0.20%`：你最多愿意买到 100×(1+0.2%)=100.2 为止。  
  这行的 `depth/notional` = 你从卖一开始一直吃到 100.2，最多能买到多少量、要花多少钱。

- `percentage=-0.20%`：你最多愿意卖到 100×(1-0.2%)=99.8 为止。  
  这行的 `depth/notional` = 你从买一开始一直卖到 99.8，最多能卖出多少量、能拿到多少钱。

直觉总结：

- 数值越大：这一侧越厚，越不容易被“打穿”。
- 买侧（负号）比卖侧（正号）厚很多：下跌更难（更像“有支撑”）。
- 卖侧厚很多：上涨更难（更像“有压制”）。

---

## 4) 怎么用（How to use）——直接落到策略/回测

### 4.1 风控门禁（最有效的第一步）

别让策略在“薄盘口”里送死：

- 如果 `notional@+0.2%` 太小：禁止用市价买（买单会把自己打成高价）。
- 如果 `notional@-0.2%` 太小：禁止用市价卖（卖单会把自己砸穿）。
- 仓位/下单金额按流动性缩放：  
  `order_notional <= notional@0.2% * k`（例如 `k=0.05`），超了就分拆/延迟/放弃。

### 4.2 冲击成本估算（回测里别再用“理想成交价”）

回测最常见的骗自己方式：假设 “成交价=last/mid”。  
更真实的做法：

- 你要买 X USDT，就看 `notional@+0.2%/+1%/+5%` 哪个刚好覆盖你，用它估算“最坏成交区间”。
- 或拟合一条“冲击曲线”当交易成本模型（对比不同时间段策略胜率差异会非常明显）。

### 4.3 盘口因子（做模型/阈值策略的最小集合）

最常用也最稳的几个：

- 不对称（imbalance）：
  - `imb_0.2 = (bid_notional@-0.2 - ask_notional@+0.2) / (bid_notional@-0.2 + ask_notional@+0.2)`
- 近端厚度（liquidity）：
  - `liq_0.2 = bid_notional@-0.2 + ask_notional@+0.2`
- 形状/斜率（近端 vs 远端）：
  - `shape = notional@0.2 / notional@1`（近端相对远端是厚还是薄）

---

## 5) 系统论视角（把它放进整套系统里看）

把市场当成一个系统：

- 订单簿 = **缓冲器 / 水库**（挂单=储备）
- 成交（trades）= **流量**（冲击把储备放掉）
- 价格 = **系统输出**（水位随储备变化而动）

`bookDepth` 这条曲线就是在问：

- **水库在不同“水位附近”（±x%）还有多少水？**

策略像控制器：

- 水库薄（流动性差）→ 降速/减仓/不追价
- 水库厚（流动性好）→ 可以更激进（但仍要控成本）

## 6) 官方下载 vs 你用 `ccxt.pro` 派生（为什么会不一样）

### 6.1 `bookTicker`：可以做到“同源同义”

- 官方 CSV 的 `update_id,best_bid_*,best_ask_*,transaction_time,event_time`
- WS `bookTicker` 原生字段 `u,b,B,a,A,T,E`

本质是同一条数据产品，不一致通常只是“格式/精度显示方式”不同。

### 6.2 `bookDepth`：列名一样，但可能不是同一个“底层输入”

官方 `bookDepth` 是 Binance 预计算的“百分比深度曲线”。  
用 `ccxt.pro` 实时你能拿到的是 L2（`watchOrderBook(limit=N)`），你只能 **自己聚合** 出“近似曲线”。

因此会有两类必然差异：

- **深度截断**：`limit=N` 看不到更深的档位，某些百分比阈值会“饱和”，出现多个 `percentage` 的 `depth/notional` 相同。
- **采样节奏不同**：官方日包是它自己的采样频率；你派生的是你自己设置的采样间隔（例如 5s 一次）。

建议：把“派生参数”（limit/interval/pcts/mid 计算方式）写入 `storage.files.meta`，避免半年后把派生曲线当官方曲线用。
